## 핵심 설계 개념 (구 ref-007/008/010/011-src에서 계승)

아래 개념은 OpenClaw 전략(폐기) 시기에 확립되었으나, 실행 방법이 바뀌어도 변하지 않는 원리다.

### 네 박자 루프 (ref-010에서 계승)

```
1. AI가 만든다      → 메모에서 관계를 자동 추출
2. diff가 드러낸다  → 과거 vs 현재 온톨로지 차이를 가시화
3. 사용자가 고친다  → Reflection에서 승인/거절/수정
4. 수정이 AI를 바꾼다 → 다음 추출에 교정 패턴 반영
```

- 이 네 박자가 한 바퀴 돌 때 개체화 한 사이클이 완료된다
- 2번(diff)을 의도적으로 설계한 제품은 아직 없다 — Meki의 차별점
- 기능은 복제 가능하지만, diff 안에 들어가는 6개월치 버전 히스토리는 복제 불가

### append-only 저장 (ref-007/008에서 계승)

- graph.jsonl은 덮어쓰기 없이 누적만 한다
- 이유: diff를 위한 버전 히스토리 보존이 목적
- Mem0도 ADD-only를 주장하지만 시스템 레벨 제약이 아님 (ref-011 검증). Meki는 구조적으로 강제

### Tool 계약 패턴 (ref-011-src에서 계승, Phase 11 이후 적용)

- 입력 스키마 검증 + 권한 체크 + 읽기/쓰기 구분 + Pre/Post Hook
- 사용자에게 "무엇이 실행됐는지"뿐 아니라 "왜 그렇게 판단됐는지" 보여줄 수 있는 구조
- Phase 11 피드백 루프 구현 시 적용

## 왜 바꿨는가

### 1. Docker 설치는 비개발자에게 벽이다

기존 계획은 사용자가 Docker Desktop을 설치하고 `docker-compose up`을 실행해야 했다. 개발자가 아닌 사람에게 이것은 사실상 불가능하다. 

Meki의 타겟은 개인이다. 팀이 아니다. 개인 사용자의 99%는 Docker가 뭔지 모른다.

### 2. 30초 온보딩이 필수다

사람들의 온라인 집중력이 없어졌다. 3단계, 30초 이내로 시작할 수 있어야 한다. GitHub OAuth는 이미 자동화되어 있으므로 원클릭이다. 여기에 Docker 설치를 끼워넣으면 온보딩이 무너진다.

### 3. 팀이 아니라 개인이다

Slack, 이메일, KakaoTalk 연동은 팀 협업 시나리오다. Meki는 개인의 관점을 발견하는 도구다. 따라서 팀 협업툴 연동을 전부 제거하고, 개인 노트앱(Apple Notes, Samsung Notes, Notion, Obsidian)만 남겼다.

### 4. 메모는 모바일 네이티브 앱에서 쓴다

메모는 input이 들어오는 그 자리에서 사람 손으로 생성된다. 보통 모바일이다. Apple Notes와 Samsung Notes의 쓰기 경험은 어떤 웹앱으로도 대체할 수 없다. Meki 자체 에디터로 메모를 쓰게 하는 것은 열등한 경험이다.

따라서 Meki는 "쓰기 도구"가 아니라 "의미 구축 도구"로 포지셔닝한다. 기존 노트앱에서 쓴 메모가 Meki로 흘러들어와서 온톨로지로 구축되는 구조.

### 5. 금광은 Apple Notes / Samsung Notes다

빈 Meki에서 시작하면 AI가 제안할 게 없다. 사용자가 이미 가지고 있는 노트 수백 개를 한 번에 가져와야 첫 "와" 순간이 만들어진다.

**핵심 관찰**: Notion 사용자는 이미 자기 나름의 정리 체계를 갖추고 있다. 폴더, 데이터베이스, 관계형 속성 — 이미 구조화된 데이터에서 Meki가 추가로 보여줄 수 있는 것이 적다. 반면 Apple Notes / Samsung Notes 사용자는 수백~수천 개의 메모를 날짜순 목록 외에 아무 구조 없이 쌓아두고 있다. 이 비구조적 메모 더미야말로 Meki가 관계와 패턴을 발견해줬을 때 가장 큰 "와우"가 나오는 곳이다.

따라서 온보딩 시 데이터 수집 우선순위를 수정했다:
1. **Apple Notes / Samsung Notes** (금광 — 비구조적 메모 더미에서 온톨로지 구축)
2. 파일 Import — 마크다운/텍스트 드래그앤드롭 (설치 없음, 범용)
3. Meki 자체 에디터 (설치 없음)
4. Notion OAuth (이미 정리된 사용자 — Meki의 핵심 타겟이 아님)

Apple Notes는 공식 클라우드 API가 없어 로컬 추출(AppleScript 또는 .mcpb)이 필요하다는 기술적 한계가 있지만, 이 마찰을 감수할 만큼 그 안의 데이터 가치가 크다. "30초 온보딩"보다 "진짜 금광에서 출발"이 더 중요하다.

### 6. 완전 자동 추출은 품질이 떨어진다

OpenClaw가 전부 자동으로 처리하는 기존 계획에서는 사용자가 구경꾼이었다. AI가 알아서 Entity/Relation을 만들고, 사용자는 결과를 보기만 한다.

문제는 자동 추출의 품질이 보장되지 않는다는 것이다. 사용자가 매일 판단(승인/거절/수정)하는 루틴이 있어야 온톨로지 품질이 보정되고, 그 과정 자체가 "내 관점 명시화"가 된다.

### 7. 온디바이스 LLM은 아직 시기상조다

Gemma4 E2B 같은 모델은 모바일에서 돌아가지만, RAM 한계로 자체 온톨로지 구축(Entity/Relation 추출 + 스키마 매핑)을 수행하기에는 아직 부족하다. 따라서 현재는 BYOK API(Gemini/Claude)로 시작하고, 온디바이스 LLM이 성숙하면 전환할 수 있도록 추상화 레이어를 둔다.

### 8. 경쟁 분석 결과

| 제품 | 교훈 |
|---|---|
| **Reflect** | 네이티브 앱 쓰기 경험 + 파일 Import로 시작. AI는 도구(요약/검색), 온톨로지 없음. 구독 모델로 수익 |
| **Mem.ai** | OAuth API로 원클릭 데이터 수집. AI가 전부 자동. diff 없음 |
| **Supermemory** | B2B 피벗. 개인용 메모리 레이어는 수익화 어려움 → 인프라 API로 전환 |
| **Saner.AI** | 크롬 익스텐션으로 실시간 캡처. 설치 허들 낮음 |

Reflect가 "처음 만들던 Meki(AI 기반 위키)"라면, 지금의 Meki는 **diff**다. 기존 제품은 ①AI가 정리해줌만 있다. ②diff(나 vs 과거의 나)를 설계한 제품은 없다.

## 재평가 및 전략 재-전환 (2026-05-07)

**"나는 사용자가 그들이 이미 익숙한 인터랙션을 바꾸도록 하지 않을 겁니다."**

### 1. 온보딩(마찰 최소화)보다 '사용자 습관 보존'이 최우선 가치다
Notion OAuth나 파일 Import가 기술적 마찰(온보딩 장벽)은 낮출 수 있습니다. 하지만 이는 사용자에게 "새로운 곳에 글을 쓰거나 데이터를 옮겨라"라고 강요하는 행위입니다. Meki의 본질은 "기존의 삶의 방식(습관) 아래에서 조용히 작동하는 구조화 도구"여야 합니다. Apple Memo와 Samsung Notes 연동은 기술적으로 폐쇄적이고 구현 난이도가 극도로 높지만, 제품의 철학을 지키기 위해 타협할 수 없는 요구사항으로 복귀했습니다.

### 2. 새롭게 재편된 진화 로드맵
- **Phase 10**: Apple Memo / Samsung Notes 연결 → AI 구조화 → 교정 UI → **개입(intervention) 저장**
  - AI가 제안한 구조화를 사용자가 어떻게 '교정'했는지 그 개입 자체가 핵심 데이터가 됩니다.
- **Phase 11**: `relationship_id` 자동 태깅 (캘린더 + 메모 맥락에서 추론) → **관계별 교정 타임라인**
  - 단순한 diff를 넘어, 시공간적 맥락(캘린더 등)을 결합해 관계의 역사를 구축합니다.
- **Phase 12**: **교정 패턴 기반 AI 시뮬레이션**
  - 축적된 교정 데이터를 통해, "과거의 나라면 이 상황에서 어떻게 생각했을까?"를 묻고 대화할 수 있는 자아의 거울(모방 에이전트)을 만듭니다.

결과적으로, Meki는 외부 세계의 정보를 단순히 정리하는 툴을 넘어, **나의 개입(판단) 패턴을 학습하는 자아 시뮬레이터**로 진화하는 방향성을 확고히 합니다.

### 3. 확정 데이터 파이프라인 아키텍처 (2026-05-12 확정)

#### 핵심 원칙
- **사용자의 능동적 행위는 "승인과 교정"뿐이다.** 추출·전송·분석은 100% 자동.
- **PC의 존재는 사용자에게 투명하다.** 사용자 경험은 "폰에서 메모 → 폰에서 Meki 열면 관계 카드가 쌓여 있다"뿐.

     │                           │                         │                   │
     │  iCloud / Samsung Cloud   │                         │                   │
     ├──── 자동 동기화 ─────────→│                         │                   │

     │                           │                         │  AI 온톨로지 추출  │

     │                           │                         │                   │ 스와이프

#### 기술 구현

**Apple Notes (Mac)**:
- MekiSync를 **Mac 메뉴바 상주 앱**(☁ 아이콘)으로 배포.
- 최초 1회 설치 후, Mac이 켜져 있는 동안 주기적으로(10분마다) JXA로 Apple Notes 스캔.
- 새로 추가/수정된 메모를 자동 감지 → Meki 서버로 전송.

**Samsung Notes (Windows)**:
- MekiSync를 **Windows 시스템 트레이 앱**(🔔 아이콘)으로 배포.
- `wdoc` 폴더에 파일 시스템 워처(File Watcher)를 설정.
- 새 `.sdocx` 파일 동기화 감지 즉시 → 오픈소스 파서(`twangodev/sdocx`)로 텍스트 추출 → Meki 서버 전송.

#### 법적 면책 방벽 (Open-source Airgap)

대기업과의 마찰을 원천 차단하기 위해, 추출 기능은 Meki 내부 코드가 아닌 **완전히 분리된 오픈소스 프로젝트**로 개발/배포한다.
- 프로젝트명: `open-notes-extractor` (MIT 라이선스)
- Meki 웹앱은 "이 오픈소스 도구로 연동하세요"라고 안내만 제공.
- Meki는 데이터 추출의 법적 책임(ToS)에서 완벽히 분리됨.

## 로컬 브릿지 앱 아키텍처 — 핵심 제약 정리 (2026-05-12 팔로업)

### 로컬 앱은 선택이 아니라 필수 조건이다

웹앱(브라우저)은 샌드박스 내에 갇혀 있어 Apple Notes, Samsung Notes의 로컬 데이터에 접근할 방법 자체가 없다. **로컬 브릿지 앱 없이는 데이터를 꺼낼 수 없다.** 이것은 설계 선택이 아니라 플랫폼 제약이다.

"Local App Bridge"(1회용 .app)와 "MekiSync 데몬"(상주 앱)은 서로 다른 접근법이 아니다. 같은 로컬 브릿지 앱의 성숙도 차이다.

| | 내용 |
|--|--|
| **카테고리** | 로컬 브릿지 앱 (유일한 방법) |
| **MVP 버전** | 1회용 — 더블클릭 → 추출 → 종료 (온보딩 대량 import) |
| **완성 버전** | 상시 가동 MekiSync 데몬 — 신규 메모 자동 동기화 |

### Phase 10 구현 순서

1. **1회용 버전 먼저**: 온보딩에서 기존 메모 수백 개 한 번에 가져옴 → "와우 순간" 확보
2. **상시 가동 버전 나중에**: 이후 신규 메모가 자동으로 흘러들어오는 구조

### 4. Reflection 매체 구조 (2026-05-12 확정)

AI가 제안하는 관계의 단위를 **"카드"가 아니라 "Reflection"**으로 명명한다.

#### 왜 Reflection인가

- "카드"는 처리하고 치우는 물건이다. "Reflection"은 나를 들여다보는 행위다.
- 이중적 의미: **반영(Reflect)** = AI가 내 메모 속 관계를 비추어 보여줌 / **성찰(Reflection)** = 그것을 보고 내가 판단하는 행위
- Phase 12의 "자아의 거울"로의 연결이 자연스럽다: Reflection의 축적이 곧 자아 시뮬레이터의 원재료.

#### 용어 체계

| 용어 | 정의 |
|---|---|
| **Reflection** | AI가 제안하는 하나의 관계 (텍스트 카드 형태) |
| **Daily Reflection** | 하루 단위 발행물. 5~7개 Reflection으로 구성 |
| **Reflection Journal** | 과거 Daily Reflection의 누적 기록 (아카이브) |
| **교정 (Intervention)** | 사용자의 판단 행위 (승인 / 거절 / 수정) |

#### 매체의 3요소

**1. 발행 기준 (Editorial Standards)**
- 비자명성: 검색으로 찾을 수 있는 자명한 연결은 발행하지 않는다
- 근거 충분성: 최소 2개 이상의 메모에서 근거가 있어야 한다
- 중복 배제: 이전 호에서 이미 처리된 관계의 변형은 발행하지 않는다
- 시의성: 최근 메모와 관련된 관계를 우선 발행한다

**2. 주기 (Periodicity)**
- 일간 발행: 매일 정해진 시점에 Daily Reflection 1건 발행
- 일일 상한 7개, 하한 1~2개 (메모 없는 날에도 과거에서 발굴)
- 첫 임포트 시 드립 피드: 2~4주에 걸쳐 분배

**3. 축적 (Archive)**
- 과거 호가 사라지지 않고 Reflection Journal로 축적
- 교정 내용의 변화 = 나의 관점의 진화 기록
- Phase 12 자아 시뮬레이터의 학습 데이터
