## 1. Prior vs Harness 구분

### Prior (사전 믿음) — "세계가 어떻게 구성되어 있는가"

온톨로지 스키마. FEP에서 관찰 이전의 세계 구조에 대한 기대.

```
Prior = {
  entityTypes:   [...],   // 세계에 무엇이 있는가
  relationTypes: [...],   // 그것들이 어떻게 연결되는가
  confidenceThresholds: { accept, review, reject }
}
```

- AI 제안 → 사용자 피드백으로 **자동 수렴** (능동추론)
- **공유했을 때 타인의 자유에너지를 줄이는 것** = 바이럴 객체
- 비유: 지도의 범례

### Harness (구속 장치) — "에이전트가 어떻게 행동해야 하는가"

에이전트의 행동 범위와 방식을 구속하는 운영 규칙.

```
Harness = {
  prompts:     { extraction, tone, summarization },
  policies:    { sources, excluded, privacy, schedule },
  permissions: { autoCreateEntity, autoCreateRelation },
  model:       { provider, temperature }
}
```

- 사용자가 **명시적으로 설정** (자동 갱신 안 됨)
- 개인 운영 환경에 종속적
- 공유 가치 낮음
- 비유: 탐험가의 규칙서

### 상위 개념: 생성모델 패키지

하니스/온톨로지는 **특정 목적과 도메인에 맞게 구성된 생성모델 패키지**.
Prior(beliefs) + Likelihood(observation mapping) + Harness(action policies).
한 공동체가 세계를 어떻게 읽고, 해석하고, 행위하는지를 캡슐화한 구조.
스티글러적 3차 파지이자 시몽동적 트랜스개체화의 핵심 매개.

## 2. Meki의 본질적 가치 (FEP 기반)

### 문제: 파편화된 생성모델

사람의 생각은 5개 앱(Apple Notes, Slack, 이메일, Notion, 메신저)에 흩어져 있다.
체감: "뭔가 연결된 것 같은데 뭔지 모르겠다", "확신이 안 선다", "같은 생각을 반복한다"
FEP 진단: 예측 오차가 해소되지 않고 축적. 자기 자신의 데이터를 자기 생성모델이 다 쓰지 못하는 상태.

### 가치: 재통합 → 예측 오차 해소

```
Meki = 파편화된 외재화 기억을 재통합하여
       자기 자신의 예측 오차를 해소하게 하는 도구
```

"노트 정리"가 아니라 **"내가 이미 알고 있는 것을 내가 쓸 수 있게 만드는 것"**.
2월 메모와 3월 대화가 연결되면 "아 그래서 확신이 안 섰던 거구나"가 해소됨.

### Prior가 추가하는 가치: 생산적 놀람

혼자서는 한계 — 내 데이터를 내 눈으로만 보기 때문.
타인의 Prior를 적용하면 **익숙한 내 데이터가 낯설어진다.**
이 낯설음(생산적 예측 오차)을 해소하는 과정이 학습이고 개체화.

```
내 데이터 (익숙함)
  → 타인의 Prior 적용 (다른 프레임)
    → 같은 데이터가 다르게 보인다 (생산적 놀람)
      → 안 맞는 곳에서 내 것이 탄생한다 (개체화)
```

## 3. Prior Fork의 본질: "저렇게 생각하고 싶다"

### 학습은 원래 이렇게 일어난다

```
철학과 학생이 들뢰즈 수업을 듣는다
  → 교수가 세계를 보는 방식에 감탄한다
  → "저렇게 읽고 싶다, 저렇게 말하고 싶다"
  → 교수의 프레임으로 텍스트를 읽어본다
  → 안 맞는 부분이 생긴다 (내 경험과 마찰)
  → 그 마찰에서 내 것이 생긴다
```

교수가 준 것: 정보가 아니라 **세계를 분절하는 방식** (Prior).
학생이 한 것: 복사가 아니라 **적용 → 마찰 → 분기** (fork → individuation).

### Fork = 동일시 → 적용 → 분기

```
fork는 "이 사람처럼 되고 싶다"에서 시작해서
"나만의 것"으로 끝나는 과정이다.
```

예시: 존경하는 투자자의 Prior를 fork
  → 내 데이터에 적용 → 그 사람의 프레임으로 재구조화
  → 안 맞는 곳 발견 (개발자인 내게는 "기술 리스크"가 보이는데 원본에 없음)
  → relation type 추가 → 원본과 다른 나만의 Prior 탄생
  → 투자자의 프레임 + 내 기술 배경 = 새로운 개체화

## 4. 바이럴의 정체: 욕망의 전파

퍼지는 것은 도구도 기능도 아니다.
**"저 사람처럼 생각하고 싶다"는 욕망이 퍼지는 것이다.**

| 영역 | 욕망 | 행위 |
|---|---|---|
| 학술 | "이 교수 밑에서 배우고 싶다" | 입학 |
| 개발 | "이 사람의 코드처럼 짜고 싶다" | star, fork |
| K-pop | "이 아티스트처럼 표현하고 싶다" | 팬덤 가입 |
| **Meki** | **"이 사람처럼 생각하고 싶다"** | **Prior fork** |

이 욕망은 **결과물을 볼 때** 발생한다.
교수의 강의, 개발자의 코드, 아티스트의 무대.
Meki에서는 그 사람이 자기 Prior로 만든 **분석, 글, 발견**을 볼 때.

### K-pop 팬덤과의 구조적 대응 (FEP 기반)

| K-pop | FEP | Meki |
|---|---|---|
| 새 앨범 발매 | 새 관찰 → 놀람 | Prior 적용 → 내 데이터가 낯설어짐 |
| 팬이 해석 | 예측 오차 해소 | 발견된 연결/모순을 수용/거절 |
| 해석 공유 | 타인의 놀람을 줄여줌 | fork된 Prior를 공유 |
| 다른 팬이 자기 해석 | 새 개체화 | 받은 Prior를 자기 데이터에 적용 → 분기 |
| 아티스트가 또 새 앨범 | 순환 지속 | **내 삶이 계속 데이터를 만든다** → 순환 지속 |

핵심 차이: K-pop에서 놀람의 원천은 아티스트(외부). Meki에서 놀람의 원천은 Prior가 내 데이터에 적용될 때 발생하는 생산적 예측 오차. 그리고 내 삶 자체가 계속 새 데이터를 생성하므로 순환이 지속된다.

## 5. 바이럴 루프: 개인에서 시작

### 최소 단위: 2명

```
나 (혼자 씀)
  → 발견이 생김 ("나 3개월째 같은 고민 반복하고 있었어")
  → 흥분해서 말하고 싶음 (자기 발견의 공유 욕구)
  → 친구에게: "야 이거 해봐"
  → 친구가 적용 → 친구에게도 발견이 생김
  → 친구가 또 다른 사람에게 말함
```

공유 동기는 이타심이 아니라 **자기 발견의 흥분**.
"야 나 이거 적용했더니 내가 2월에 SSE 쓰겠다고 해놓고 3월에 WS로 바꾼 거 잡혔어 ㅋㅋ"

### 그룹 간 전파: 겹치는 사람이 fork한다

```
민수 (사이드프로젝트 Prior v1.0)
  → 팀 3명에 공유 → Prior v1.1로 진화

지연 (팀원이면서 독서 모임 멤버)
  → Prior를 fork → 도메인 어휘만 변경
     Tool→Thinker, Problem→Concept, Decision→Claim
  → 구조적 뼈대(contradicts 등)는 유지
  → 독서 모임 3명에 공유
```

fork의 본질: **구조적 뼈대는 유지하면서 도메인 어휘만 바꾸는 것.**
"무언가가 무언가를 다루고, 선택이 생기고, 선택 간 긴장이 있다" — 이 패턴은 도메인을 넘어 보편적.

### MCP가 cold start를 해결

Prior만 받으면 적용할 데이터가 없어서 가치 = 0.
MCP로 기존 도구(Apple Notes, Slack, 이메일)를 즉시 연결 → 마이그레이션 불필요 → 설치 즉시 가치 발생.

## 6. 공동체 설계: 지지체 + 인용 규범

### 실제 공동체의 공통 구조

**특정 생성모델을 공유하는 사람들이, 그 모델의 예측을 집단적으로 실현하면서, 모델 자체를 함께 갱신한다.**

### Meki의 지지체: Prior Registry

```
Prior Registry
├── 발행: "투자분석 v2.3" by @halim (forks: 47, applied: 312)
├── 계보: @halim ← @kim/가치투자 ← @park/기업분석-기초
├── 기여: v2.3에 "regulatory_risk" 추가 (제안: @lee)
└── 진화: v1.0→v1.5→v2.0→v2.3 타임라인
```

### Meki의 인용 규범: Provenance Chain

1. Fork 귀속 — 지적 계보 추적
2. 기여 크레딧 — relation type 제안자 이름 보존
3. 진화 근거 — 변경의 WHY 기록
4. 비호환 선언 — 충돌하는 Prior 간 브리지 필요성 명시

### Confidence Aggregation: 데이터 없이 집단 학습

312명이 같은 Prior를 적용 → 각자 승인/거절률만 공유 (내용 비공유)
→ 집계 결과로 Prior의 confidence default가 갱신
= 연합 학습의 Prior 버전. 데이터 주권 유지하면서 집단적으로 Prior가 정밀해짐.

### Meki의 플랫폼 정체성: 개체화의 장

Meki는 세 플랫폼의 구조를 통합한다.

```
arXiv   → 공개 지식, 인용(provenance), 누구나 접근
GitHub  → fork/contribute, 계보 추적, 기여 크레딧
Twitter → 팔로워 기반 영향력, 자연 바이럴

Meki    → 위 세 개 + "내 데이터에 직접 적용"이라는 능동성
         = 개체화가 일어나는 인프라
```

### 수익 모델: Provenance 기본, Sponsors 선택

**원칙: Prior fork는 항상 무료. 돈은 자발적 후원으로만 흐른다.**

```
1단계 — 기본 (모든 크리에이터):
  Prior 공개 → 누구나 fork → Provenance chain 자동 기록
  (arXiv 인용처럼 — 무료, 강제 귀속)

2단계 — Sponsors 활성화 (팔로워가 쌓인 크리에이터 선택):
  "이 Prior로 개체화됐다, 후원하고 싶다" →
  사용자가 자발적으로 후원 금액 설정
  (GitHub Sponsors처럼 — 강제 아님, 감사 기반)

3단계 — 팀 플랜 (조직이 Prior를 함께 운영):
  팀 Confidence Aggregation + 권한 관리 + 감사 로그
  → Meki 월정액
```

**fork된 Prior의 수익 귀속:**
- 기본: Provenance만 (계보 기록, 금전 흐름 없음)
- 선택: 사용자가 후원 시 "원본 크리에이터에게도 일부" 설정 가능
  - 예: 후원 $5 중 $1을 upstream Prior 원작자에게
  - 강제 아님 — 개체화된 fork는 독립적이므로 의무 없음

**Meki의 수익:**
```
후원 수수료 (Sponsors 트랜잭션의 일정 %)
+ 팀 플랜 월정액
+ 선택적 compute 호스팅 (자체 API 키 없는 사용자)
```

### 왜 이 모델인가

- **paywall = 바이럴 사망**: Prior에 가격표를 붙이면 fork가 멈춤
- **후원은 감사**: "이 사람의 관점이 내 개체화에 기여했다" → 자발적 흐름
- **Provenance = 장기 신뢰**: fork 횟수, 적용 횟수, 계보 깊이가 크리에이터의 신뢰 자산
- **팔로워 → Sponsors**: 자연스러운 성장 경로. 무명에서 시작해 영향력이 생기면 수익 전환

### 바이럴

Prior 공개 → fork → 적용 → "내 데이터가 다르게 보인다" 체험 → 공유 →
다음 사람이 "나도 해보고 싶다" → 설치 → fork.

크리에이터가 유통 채널. Meki가 직접 바이럴할 필요 없음.

### 경쟁 우위 vs NotebookLM / ChatGPT

```
NotebookLM: 크리에이터 콘텐츠를 소비 (질문→답변, 블랙박스)
ChatGPT:    스냅샷 분석 (대화 끝나면 사라짐)
Meki:       크리에이터 관점에 참여 (fork→적용→개체화→공동체)
```

블랙박스는 공동체를 만들지 못한다.
명시적이고 fork 가능한 Prior만이 Provenance chain을 만들고, chain이 공동체가 된다.

### 모트(moat)

```
복제 가능: 기능 (AI 추출, 그래프, MCP 연결)
복제 불가: Provenance chain에 축적된 개체화 계보
           — 빅테크가 기능을 복제해도 3년간 쌓인 fork 계보는 복제 불가
```

## 10. 한국 시장 인사이트 (read.md 발췌)

- **OpenClaw 실재 확인**: 2026-03 기준 중국 전역 사회 현상. 100일 만에 17만 GitHub stars.
- **아키텍처 교훈**: API/MCP 기반(OpenClaw) > GUI 에이전트(Doubao 폰). 개방형이 이김.
- **한국 경로**: 빅테크 탑다운 불가 → 스타트업 바텀업, 버티컬 우선
- **대안5가 유력**: 소셜 기능 + 에이전트 기능 하이브리드 → 바이럴 + 밸류 캡처 동시 해결
- **병목은 기술이 아니라 구조**: 폐쇄적 플랫폼(네이버, 쿠팡)과 AI 에이전트 사이의 통합 격차
- **기능은 복제 가능, WHY는 복제 불가**: 개인 크리에이터는 서사(narrative) 경쟁에서 빅테크를 이김
