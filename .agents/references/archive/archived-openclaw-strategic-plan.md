# Meki Phase 10~11 개발 플랜

## 핵심 전제: 대화에서 도출된 전략 전환

### Meki는 무엇인가

Meki는 대시보드가 아니다. 노트앱도 아니다.
**에이전트가 세계를 어떻게 이해하고 있는지 볼 수 있는 창(window into agent's world model)**이다.

agent.md는 입력(이렇게 행동해라)이다.
Meki는 출력(에이전트가 세계를 이렇게 이해하고 있다)을 보여준다.

### Git 비유: 포지셔닝의 본질

Git 이전에도 `cp project_v1/ project_v2/`로 버전관리가 가능했다.
Git이 한 건 이론적으로 가능한 걸 **자동으로 항상 일어나게** 만든 것이다.

agent.md에 "사고과정을 기록하라, 온톨로지를 출력하라, diff를 보여달라"고 적을 수 있다.
하지만 아무도 안 적는다. 적더라도 매주 쌓이는 텍스트 로그를 비교하는 사람은 없다.

**Meki = 가능하지만 아무도 안 하는 걸 자동으로 일어나게 만드는 인프라**

### 네 박자 루프: 개체화 사이클

```
1. AI가 만든다     → OpenClaw가 컨텍스트에서 온톨로지 자동 구축
2. diff가 드러낸다 → 과거 vs 현재 온톨로지의 차이를 가시화
3. 사용자가 고친다 → 리뷰 UI에서 온톨로지 수정
4. 수정이 AI를 바꾼다 → 다음 OpenClaw 실행에 반영
```

이 네 박자가 한 바퀴 돌 때 개체화 한 사이클이 완료된다.
기존 제품(Mem0, Glean)은 1번만, 학술 연구(SCHEMA-MINERpro)는 1+3+4만,
**2번(diff)을 의도적으로 설계한 제품은 아직 없다.**

### 생존 구조

기능은 복제 가능하다. diff UI는 하루 만에 만든다.
하지만 diff 안에 들어가는 6개월치 온톨로지 버전 히스토리는 복제 불가능하다.

**기능이 아니라 시간이 해자(moat)이다.**

따라서: 빨리 시작하게 만들어서, 경쟁자가 복제할 때쯤 사용자에게 이미 대체 불가능한 시간이 쌓여 있게 한다.

### 유통 채널

별도 마케팅이 아니라 **OpenClaw 자체가 유통 채널**이다.
OpenClaw 사용자가 "내 에이전트가 왜 이렇게 행동했지?"를 물을 때 답이 없다.
Meki가 OpenClaw의 기본 레이어로 들어가면 별도 선택이 필요 없다.

## Phase 10: 감각 기관 — "AI가 만든다"

**한 줄**: MCP로 외부 도구 연결 → OpenClaw가 자동으로 의미 그래프 구축

### 마일스톤

| # | 마일스톤 | 완료 기준 | 의존성 |
|---|---------|-----------|--------|
| M10.1 | Docker 로컬 환경 구축 | OpenClaw 로컬 실행, 외부 네트워크 전송 없음 확인 | — |
| M10.2 | MCP 게이트웨이 | 1개 이상 외부 소스(Notion or Slack) 연결 성공 | M10.1 |
| M10.3 | 온톨로지 자동 구축 | 외부 데이터에서 Entity/Relation 최소 10개 자동 생성 | M10.2 |
| M10.4 | 버전 저장 시작 | graph.jsonl에 타임스탬프 포함 append-only 기록 | M10.3 |
| M10.5 | 그래프 뷰어 | 생성된 온톨로지를 시각적으로 확인 가능 | M10.3 |

### 핵심 설계 원칙

- **M10.4가 가장 중요하다.** diff는 Phase 11이지만, diff의 원재료(버전 히스토리)는 Phase 10부터 쌓여야 한다. 나중에 추가하면 과거 데이터가 없다.
- append-only + 타임스탬프 구조는 Phase 11의 스냅샷 diff를 자연 지원한다.
- 이 시점에서 사용자 가치: "내 Notion/Slack 데이터를 AI가 정리해줬다" (생산성)

### Phase 10 사용자 경험

```
사용자: OpenClaw + Meki 설치 (Docker)
       → Notion MCP 연결
       → OpenClaw가 Notion 페이지를 읽고 온톨로지 생성
       → Meki 뷰어에서 "아, 내 데이터가 이렇게 연결되는구나" 확인
```

## Phase 11: 개체화 인터페이스 — "diff가 드러내고, 사용자가 고치고, AI가 바뀐다"

**한 줄**: 온톨로지 리뷰·수정 + 버전 diff → 네 박자 루프 완성

### 마일스톤

| # | 마일스톤 | 완료 기준 | 의존성 |
|---|---------|-----------|--------|
| M11.1 | 리뷰 UI | Entity/Relation 승인·거절·수정 인터페이스 | M10.3 |
| M11.2 | 하니스 에디터 | HarnessProfile 편집 가능 (prompts, policies, schema) | M10.3 |
| M11.3 | 피드백 파이프라인 | 사용자 수정 → 다음 OpenClaw 실행에 반영 확인 | M11.1 |
| M11.4 | 버전 diff 엔진 | 두 시점의 온톨로지 스냅샷 구조적 비교 | M10.4 |
| M11.5 | diff 시각화 | 동일 입력에 대한 두 하니스 버전의 출력 차이 가시화 | M11.4 |
| M11.6 | 루프 검증 | 수정 → 재실행 → diff 확인까지 전체 사이클 1회 완료 | M11.3 + M11.5 |

### M11.4~M11.5: diff의 세 층위

1. **스키마 diff**: 두 시점의 schema.yaml 구조 비교
   - "3월에는 '반박함' 관계가 없었는데, 4월에 생겼다"
   - 가장 구현이 쉽고, 가장 먼저 보여줄 수 있다

2. **온톨로지 diff**: 같은 스키마 내에서 Entity/Relation 변화
   - "3월에는 A→B를 '원인'으로 봤는데, 4월에는 '결과'로 바뀌었다"
   - graph.jsonl의 타임스탬프 기반 필터링

3. **행동 diff** (Phase 11 후반 또는 Phase 12 초기):
   - 같은 문서를 과거 하니스 vs 현재 하니스로 처리했을 때 출력 비교
   - "같은 기사인데 3월의 나는 원인 구조를, 지금의 나는 결과 구조를 뽑는다"

### 핵심 설계 원칙

- **diff는 요청이 아니라 자동이어야 한다.** "비교하기" 버튼을 누르는 게 아니라, 일정 기간이 지나면 자동으로 변화 요약이 뜬다. Git의 `git log`가 항상 거기 있듯이.
- **diff의 첫 경험이 개체화 촉발 지점이다.** "어, 나 이렇게 달라졌네?"가 Meki가 도구에서 거울이 되는 순간이다. 이 순간의 UX가 전체 제품의 리텐션을 결정한다.
- **Phase 11에서는 "나 vs 과거의 나"만으로 충분하다.** "나 vs 남"은 Phase 12. 개체화의 최초 촉발은 타인과의 차이가 아니라 자기 자신과의 차이에서 온다.

### Phase 11 사용자 경험

```
[3개월 사용 후]
사용자: Meki 열기
       → "지난 3개월간 온톨로지 변화 요약" 자동 표시
       → "이전에는 'AI 위기'를 기술 문제로 분류했는데,
          지금은 사회 구조 문제로 분류하고 있습니다"
       → 사용자: "어, 내가 이렇게 달라졌구나"
       → 일부 수정: "이건 의도한 변화가 아니야" → 수정
       → 다음 OpenClaw 실행에 반영
       → 다음 달 diff에서 수정 효과 확인
```

```
├── Week 5-6: M11.4~M11.5 (diff 엔진 + 시각화)

총 10~14주
```

## 성공 지표

### Phase 10 (도구로서의 가치)
- OpenClaw 사용자 중 Meki 뷰어 접근율
- 자동 생성 Entity/Relation 수와 정확도

### Phase 11 (거울로서의 가치)
- 리뷰 UI에서 사용자가 실제로 수정하는 비율 (루프가 돌고 있는가)
- diff 화면 체류 시간 (diff가 의미 있는 놀라움을 주는가)
- diff 확인 후 하니스 수정으로 이어지는 전환율 (개체화가 촉발되는가)

## 열린 질문 (Phase 12 진입 전 결정 필요)

1. **diff 주기**: 자동 diff 요약은 주 단위? 월 단위? 사용자 설정?
2. **Phase 12 진입 조건**: "나 vs 과거의 나" diff가 충분히 동작한 뒤에야 "나 vs 남" 교환이 의미 있다. 몇 명의 사용자가 몇 사이클의 루프를 돌아야 Phase 12가 필요해지나?
3. **OpenClaw 통합 깊이**: Meki가 OpenClaw의 공식 대시보드가 되는가, 아니면 선택적 플러그인인가? 전자가 유통에 유리하지만 OpenClaw 팀과의 합의가 필요하다.
