## Phase 10: MCP 게이트웨이 + OpenClaw 통합

**한 줄 요약**: 기존 도구(Notion, Slack 등)를 MCP로 연결 → OpenClaw가 자동으로 의미 그래프 구축

### 핵심 설계 결정

| 결정 | 이유 |
|------|------|
| OpenClaw를 Docker로 로컬 실행 | 데이터 주권 — 모든 데이터가 사용자 기기에만 존재 |
| OpenClaw Ontology Skill 그대로 활용 | Entity/Relation/Schema 구조가 Meki 요구사항과 동일, 바퀴 재발명 불필요 |
| MCP 게이트웨이 별도 서비스 | Notion/Slack/이메일마다 다른 인증 흐름, 게이트웨이로 추상화 |
| graph.jsonl + schema.yaml 저장 | append-only 구조가 프로비넌스 추적 자연 지원 |
| 기존 KnowledgeGraph.jsx 확장 | 완전 신규 구현 불필요, 60% 기반 이미 존재 |

### 태스크 순서 (의존성)
```
P10-T1 (Docker 환경) → P10-T2 (MCP 게이트웨이) → P10-T3 (Ontology 파이프라인)
                                                  → P10-T4 (Graph RAG 인덱싱)
P10-T3 + P10-T4 완료 후 → P10-T5 (뷰어 UI) → P10-T6 (검증)
```

### 완료 기준
- MCP로 1개 이상 외부 소스 연결 성공
- OpenClaw가 자동으로 Entity/Relation 생성 (최소 10개 이상)
- 뷰어에서 그래프 확인 가능
- 데이터가 로컬에만 저장됨 (네트워크 전송 없음)

## Phase 11: 하니스 리뷰 + 피드백 루프

**한 줄 요약**: AI가 만든 온톨로지를 사용자가 리뷰·수정 → 에이전트 행동이 수렴

### 핵심 설계 결정

| 결정 | 이유 |
|------|------|
| 승인/거절/수정 인터페이스 | 완전 자동은 품질 저하, 완전 수동은 부담 — 중간 피드백 루프가 최적 |
| HarnessProfile 스키마 명시화 | `{ name, ontologySchema, prompts[], policies[], activeModel, version }` — 버전관리 가능한 단위 |
| 피드백 → 에이전트 재실행 파이프라인 | 사용자 수정이 다음 OpenClaw 실행에 반영되어야 루프 완성 |
| 온톨로지 버전 스냅샷 | diff 비교 없이는 "무엇이 어떻게 개선됐는지" 사용자가 인지 불가 |

### 철학적 의미
하니스 조정 = FEP의 생성모델 파라미터 갱신 = 시몽동의 개체화 사건.
"사용자가 AI를 고치는 것이 아니라, AI와의 대화를 통해 자신의 의미체계를 명시화하는 과정."

### 태스크 순서
```
P11-T1 (리뷰 UI) + P11-T2 (하니스 에디터) — 병렬
→ P11-T3 (피드백 파이프라인) — P11-T1 완료 후
→ P11-T4 (버전 스냅샷) — P11-T3 완료 후
→ P11-T5 (검증)
```

### 완료 기준
- 사용자가 AI 온톨로지 리뷰·수정 가능
- 하니스 편집 후 에이전트 재실행 시 행동 변화 확인
- 온톨로지 버전 diff 비교 가능

## Phase 12: 공동체 교환 인프라

**한 줄 요약**: 개인 하니스/온톨로지를 fork·merge·share 가능한 공공 자산으로

### 핵심 설계 결정

| 결정 | 이유 |
|------|------|
| GitHub 또는 Claw Hub 활용 | 전용 플랫폼 신규 구축 대신 기존 생태계 활용 |
| 브리지 온톨로지 지원 | 유크 후이 코스모테크닉스 — 단일 메타온톨로지 강제 금지, 이종 온톨로지 공존 |
| 프로비넌스 체인 | 스티글러 — 집단 기억은 기여자와 맥락이 추적 가능해야 의미 있음 |
| fork/PR 워크플로 | GitHub 개발자 공동체 패턴 차용 — 이미 검증된 집단 개체화 모델 |

### 완료 기준
- 하니스 공개·fork·PR 워크플로 동작
- 브리지 온톨로지로 이종 온톨로지 간 개념 참조 가능
- 기여 이력 프로비넌스 추적

## Meki 차별화 요약

```
OpenClaw = 엔진 (범용 에이전트)
Meki     = 조종석 (하니스 에디터 + 온톨로지 리뷰 + 버전관리 + 커뮤니티)
```

OpenClaw로 대체할 수 없는 Meki의 핵심:
1. 비개발자도 자기 의미체계를 이해·조정할 수 있는 UX
2. 개체화 인프라라는 철학적 정체성 (개인 → 공동체 연결)
3. 데이터 주권 보장 구조 (로컬-first, E2EE)
