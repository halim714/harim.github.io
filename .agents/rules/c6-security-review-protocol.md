# 보안 코드 적대적 자기검증 (Adversarial Self-Review)

## 컨텍스트
에이전트가 보안 코드를 작성할 때 일관되게 나타나는 실패 패턴:
1. **폴백 = 우회**: 범용 개발의 "graceful fallback"을 보안 코드에 적용 → 우회 경로 생성
2. **작동 = 정확**: 스크립트가 에러 없이 실행되면 올바르다고 판단 → 거짓 PASS 반환
3. **부분 = 전체**: 개별 체크가 통과하면 E2E 흐름도 안전하다고 가정 → 토큰이 여전히 평문 저장

## 규칙

### 보안 관련 코드를 작성/수정한 에이전트의 의무

1. **적대적 자기검증 (Self Red-Team)**:
   보안 코드를 작성한 후, 자신이 작성한 체크를 **우회하는 방법을 최소 1개 찾아서 기록**하라.
   우회 방법을 찾았으면 → 해당 우회를 차단하는 코드를 추가하라.
   우회 방법을 못 찾았으면 → `.agents/progress/`에 "우회 시도 0건"을 명시하라.

   ```
   # 예: CORS 검증 스크립트 작성 후
   에이전트 자기질문: "이 체크를 통과하면서 실제로는 안전하지 않은 상태가 가능한가?"
   → referer 폴백이 있으면 origin 없이도 통과 가능 → 폴백 제거
   → 주석에 "ghToken"이 있으면 payload 체크가 오탐 → 주석 strip 추가
   ```

2. **E2E 흐름 기반 사고 (End-to-End Thinking)**:
   개별 파일이 아닌, **토큰이 태어나서 죽는 전체 경로**를 기준으로 검증하라.
   "A 파일은 안전하다"가 아니라 "A → B → C 전체 경로에서 토큰이 노출되는 지점이 있는가?"를 확인하라.

   ```
   # 예: HttpOnly 쿠키 구현 후
   ❌ "server.js에 res.cookie() 추가 완료" → 부분 검증
   ✅ "callback.js → auth.js → localStorage 경로가 여전히 존재하므로,
       쿠키와 무관하게 토큰이 평문 저장됨" → E2E 검증
   ```

3. **보안 코드에 폴백 금지 (No Fallback in Security)**:
   보안 검증 로직에 `|| fallback` 패턴을 사용하지 마라.
   "없으면 빈 값으로"가 아니라 "없으면 즉시 거부"가 올바른 보안 패턴.

   ```javascript
   // ❌ 보안 안티패턴
   const origin = req.headers.origin || req.headers.referer || '';
   // ✅ 보안 패턴
   const origin = req.headers.origin || '';
   ```

### 오케스트레이터의 의무

4. **보안 태스크 완료 후 security-state-check.sh 반드시 실행**:
   보안 관련 코드를 수정하는 태스크가 완료되면, 커밋 전에 `security-state-check.sh`를 실행하고
   exit code가 0이 아니면 커밋하지 마라.

5. **거짓 PASS 테스트**:
   보안 체크 스크립트를 신규 작성하거나 수정한 후에는,
   **의도적으로 잘못된 코드를 임시 삽입하여 스크립트가 FAIL을 반환하는지** 검증하라.
   (테스트 후 원복)
