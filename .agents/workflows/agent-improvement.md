# Agent Improvement Protocol

## 절대 금지 규칙

❌ 추측으로 원인을 확정하지 마라 — 변수 격리 테스트 먼저
❌ 한 번의 비교로 인과관계를 주장하지 마라
❌ 자기가 만든 산출물을 검증 없이 PASS 판정하지 마라

> **핵심 원칙**: 코드를 고치는 것은 증상 치료다. 룰과 워크플로우를 고치는 것이 근본 치료다.

---

## 언제 이 워크플로우를 실행하는가

- 스웜 에이전트가 동일한 유형의 오류를 반복할 때
- 생성된 코드가 Meki의 핵심 가치(데이터 주권, 사유 흐름, 위키 연결성)에 위배될 때
- 에이전트가 잘못된 가정으로 작업을 진행했을 때
- `swarm-manager` 워크플로우 Step 4(검증)에서 문제가 발견되었을 때

---

## Step 1: 진단 (Root Cause Analysis)

세 가지 질문을 순서대로 답하라.

### Q1. 에이전트가 실행됐는가?
`.raw` 파일이 0바이트이거나 에이전트가 시작조차 안 됐다면:
→ **C7: 환경/초기화 실패** — `c7-hang-diagnosis.md` bisect 절차 실행

### Q2. 산출물이 기능하는가?
에이전트가 실행됐으나 결과물이 잘못됐다면:
→ **부록의 실패 패턴(C1~C5)을 참조하여 원인을 자연어로 분석하라.**

> **C1 vs C5 구분**: 컨텍스트 문서가 존재했는데 에이전트가 몰랐다 → C1. 문서 자체가 없었다 → C5.
> **C4 책임 소재**: SOP에 검증 단계가 있었는데 건너뛴 것 → 에이전트 실패(role 파일 강화). SOP에 검증 단계 자체가 없었던 것 → 워크플로우 설계 실패(swarm-manager 수정).

### Q3. 산출물이 보안/정확성 기준을 만족하는가?
기능은 하지만 보안 또는 정확성이 틀렸다면:
→ **C6: 보안 정확성 오류** — "작동함 ≠ 올바름". `c6-security-review-protocol.md` 참조

---

## Step 2: 하니스 수정 (Harness Fix)

분류 결과에 따라 룰 또는 워크플로우를 수정하라.

**룰 수정** (C1~C7): `.agents/rules/` 에 새 파일 생성 또는 기존 파일 업데이트

**파일 네이밍**: `{분류코드}-{주제}.md` (예: `c1-github-api-constraints.md`)

```markdown
---
trigger: always_on
description: [이 룰이 방지하는 실패 유형 한 줄 요약]
---

# [룰 제목]

## 컨텍스트
[왜 이 룰이 필요한지 — 어떤 실패에서 비롯되었는지]

## 규칙
- [구체적이고 행동 가능한 지시 1]
- [구체적이고 행동 가능한 지시 2]

## 올바른 예시 / 잘못된 예시
```

**워크플로우 수정** (C3, C5, C4의 워크플로우 책임 경로): `swarm-manager.md` 또는 해당 워크플로우 파일 직접 수정

**Layer 2/3 판단**: 룰/워크플로우 수정 후 다음 질문을 하라:
> "이 실패는 미래에 자동 감지 가능한가?"
> → Yes: 감지 스크립트(`scripts/`) 또는 런타임 가드(`run-swarm.sh`) 생성/업데이트
> → No: 룰만으로 충분

---

## Step 3: 교차 검토 (조건부)

**게이트 — 산출물 유형에 따라 적용 여부 결정:**

```
자동 검증 가능 (빌드/테스트 통과로 확인 가능) → 스킵
자동 검증 불가 (룰 파일, 워크플로우 문서)     → 적용
```

**적용 시**: 다른 Claude Code 세션에 산출물을 전달하여 아래 질문을 묻는다:
> "이 수정을 우회하거나 무효화하는 방법이 있는가?"

같은 세션의 자기검토는 확증 편향을 극복하지 못한다. 다른 관점이 필요하다.

**의무 아님**: 시간/비용 제약 시 생략 가능. 생략 시 Step 5 "검증 결과"에 명시.

---

## Step 4: 재실행 (Self-Correction)

Step 2 완료 후 (Step 3 해당 시 통과 후) 실패했던 태스크를 재실행:

```bash
./scripts/run-swarm.sh <모델> "<원래 실패한 태스크 프롬프트 + 새 룰 파일명 명시>" /Users/halim/Desktop/meeki/meki/miki-editor <식별자>-retry
```

---

## Step 5: 보고 (Verification & Report)

재실행 후 사용자에게 보고:

```
**무엇을 고쳤는가**: [파일명 + 한 줄 설명]
**왜 고쳤는가**: [실패 원인 자연어 서술 — 분류 강제 없음]
**검증 결과**: [빌드/테스트 결과, Step 3 결과 또는 생략 여부, C7 bisect 결과 등]
```

## [부록] 실패 패턴 참조

> 진단의 출발점이지 보고서의 필수 필드가 아니다. 실제 실패는 복합 원인인 경우가 많다.

| 분류 | 설명 | 수리 대상 |
|---|---|---|
| **C1: 컨텍스트 부재** | 에이전트가 프로젝트 구조나 제약을 몰랐음 | role SOP / 특정 context doc |
| **C2: 가치 위반** | Meki 핵심 가치에 반하는 구현 | `CLAUDE.md` / `mekirule.md` |
| **C3: 스코프 혼동** | 병렬 에이전트 간 파일 충돌 | `swarm-manager.md` + worktree 규칙 |
| **C4: 검증 부재** | 에이전트가 자기검증 없이 완료 보고 | role SOP 완료 체크리스트 또는 워크플로우 |
| **C5: 프롬프트 모호** | 지시가 넓어 에이전트가 추측으로 구현 | `swarm-manager.md` 프롬프트 템플릿 |
| **C6: 보안 정확성 오류** | 기능은 하지만 보안/정확성이 틀림 | 해당 구현 + `security-state-check.sh` |
| **C7: 환경/초기화 실패** | 에이전트가 시작조차 안 됨 | `run-swarm.sh` + `c7-hang-diagnosis.md` |
